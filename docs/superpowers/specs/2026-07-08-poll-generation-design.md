# Опросы: ИИ создаёт Telegram-опрос по запросу владельца

Дата: 2026-07-08

## Проблема

Сейчас бот умеет генерировать только текстовые/медиа-посты. Владелец хочет,
чтобы ИИ мог по запросу подготовить нативный Telegram-опрос (`bot.send_poll`)
для канала — либо как самостоятельный элемент календаря, либо как дополнение
к обычному посту. Публикация опроса должна подчиняться главному правилу
проекта: **никогда не публикуется без явного одобрения владельца**.

## Сценарии использования

1. **Самостоятельный опрос** — владелец добавляет опрос в календарь так же,
   как обычный пост, без привязки к какому-либо тексту.
2. **Опрос к посту** — владелец хочет, чтобы опрос сопровождал конкретный
   пост. Это два независимых элемента календаря, связанных для контекста;
   у каждого — своё одобрение и своё время публикации (не одно общее
   действие).

## Архитектура

Опрос — это **не новая сущность**, а `ContentTask` с `task_type="poll"`.
Это даёт опросу бесплатно весь существующий пайплайн: расписание
(`scheduler`), наводящие вопросы владельцу, генерацию черновика с версиями
и правками, одобрение (`approval.py`), аудит (`audit.py`), рассылку
уведомлений всем активным админам (`notify.broadcast`) — без дублирования
этой логики в отдельном модуле.

### Изменения модели данных

`app/database/models.py`, `ContentTask`:

- `task_type: Mapped[str]` — новый `str`-enum `TaskType` (по образцу уже
  существующих `TaskStatus`/`MediaType`/`UserRole`): `POST = "post"` (default,
  поведение не меняется) | `POLL = "poll"`. Новая миграция Alembic
  (`0006_poll_task.py`).
- `related_task_id: Mapped[int | None]` — self-FK на `content_tasks.id`,
  nullable. Необязательная связь «опрос к посту N» — только для отображения
  в Mini App («🗳 к посту #N «тема»»), не влияет на одобрение/публикацию.

Текст вопроса и варианты ответа **не получают отдельных колонок**.
Переиспользуется существующая таблица версий `GeneratedPost`
(`task.posts[-1].text`) — так опрос бесплатно получает версионирование,
«другой вариант» и правки, которые уже реализованы для постов.

**Формат сериализации** вопроса/вариантов в `GeneratedPost.text`:
- 1-я непустая строка — вопрос.
- Каждая следующая непустая строка — один вариант ответа (2–10 штук,
  ограничение Telegram: вопрос ≤300 символов, вариант ≤100 символов).

Настройки опроса (`is_anonymous`, `allows_multiple_answers`) владельцем не
конфигурируются — фиксированные значения `is_anonymous=True`,
`allows_multiple_answers=False` (обычные Telegram-дефолты для канальных
опросов). Расширять до конфигурируемых — отдельная задача, если понадобится.

### Точки входа (создание задачи-опроса)

Обе точки входа создают `ContentTask(task_type="poll", status="scheduled", ...)`
и дальше идут по тому же пути, что и обычный пост (`ask_questions` →
`generate_from_answers` → `WAITING_FOR_APPROVAL`).

1. **Текстовая метка в массовом добавлении** (Mini App, textarea `#bulk`,
   эндпоинт `POST /tasks/bulk`, парсер `content_tasks.bulk_create_tasks` /
   `parse_schedule_line`). Строка с меткой перед темой создаёт опрос вместо
   поста:
   ```
   05-07-2026 — 📊 Какой формат вам интереснее?
   06-07-2026 10:00 — Разбор ошибок
   ```
   Первая строка — опрос (тема после метки становится темой опроса), вторая —
   обычный пост. Можно свободно мешать в одном списке.

2. **Кнопка «+ Опрос» в Mini App** — компактная форма (тема + дата/время) для
   добавления одного опроса без набора синтаксиса строки. Отправляет на тот
   же эндпоинт `/tasks/bulk` одной строкой с меткой опроса (переиспользует
   парсер, не дублирует логику создания).

### Генерация черновика опроса

`app/ai/prompts.py` — новая функция `build_poll_prompt(task)`: просит ИИ
сформулировать вопрос и 2–5 коротких вариантов ответа по теме, каждый вариант
с новой строки, без нумерации и пояснений (стилистически как уже существующий
`build_generation_prompt` для наводящих вопросов).

`app/services/content_tasks.py`, `generate_post_version()` — ветка по
`task.task_type`:
- `"post"` — как сейчас, без изменений.
- `"poll"` — вызывает `build_poll_prompt`, результат парсится и **сразу
  валидируется**: 2–10 вариантов, лимиты длины Telegram. Если ИИ вернул
  некорректный опрос (например, один вариант или пустой вопрос) — это
  трактуется как сбой генерации, аналогично недоступности AI-провайдера:
  статус задачи не переводится в `WAITING_FOR_APPROVAL`, владельцу уходит
  то же предупреждение об ошибке генерации, можно запросить повторную
  попытку. Никакой битый черновик не сохраняется как готовый к одобрению.

### Публикация

`app/services/publishing.py`, `publish_task()` — ветка по `task.task_type`:
- `"post"` — как сейчас (текст/медиа, логика цитат из `_render_quote`).
- `"poll"` — парсит `task.posts[-1].text` в (question, options), вызывает
  `bot.send_poll(chat_id=channel_id, question=question, options=options,
  is_anonymous=True, allows_multiple_answers=False)`. Ошибки Telegram API
  оборачиваются в `PublishError` с понятным текстом владельцу — по аналогии
  с уже существующей проверкой `_check_bot_is_admin`.

Логика цитат (`is_quote`/`quote_text`) к опросам не применяется — эти поля
просто остаются неиспользуемыми для `task_type="poll"`.

### Mini App

`app/webapp/routes.py` — сериализация задачи включает `task_type` и (если
есть) `related_task_id` с темой связанного поста.

`app/webapp/templates/app.html` — карточка задачи с `task_type=="poll"`:
- Показывает вопрос и варианты ответа как read-only превью (без UI цитаты
  и медиа — они неприменимы к опросу).
- Те же кнопки, что и у поста: «✅ Одобряю» / «🔁 Другой вариант» /
  «✏️ Правки» / «📢 Опубликовать сейчас» — используют существующие
  эндпоинты без изменений, отдельной логики одобрения для опросов нет.
- Если задан `related_task_id` — пометка «🗳 к посту #N «тема»».
- В подписи к textarea `#bulk` — краткая подсказка про метку опроса.
- Кнопка «+ Опрос» рядом с массовым добавлением (см. точки входа выше).

## Обработка ошибок

- Некорректный ответ ИИ при генерации опроса → не сохраняется как черновик,
  трактуется как сбой генерации (см. «Генерация черновика опроса» выше).
- Ошибка Telegram API при `bot.send_poll` (например, канал не поддерживает
  опросы) → `PublishError` с понятным текстом, задача не помечается
  опубликованной (существующий механизм `PublishError` в `publish_task`).
- Некорректная строка с меткой опроса в массовом добавлении (нет темы после
  метки) → попадает в список `errors`, как и любая другая нераспознанная
  строка сейчас.

## Тестирование

- Юнит-тест парсинга опроса из ответа ИИ: валидные 2–10 вариантов проходят,
  1 вариант или отсутствие вопроса — отклоняются.
- Юнит-тест `bulk_create_tasks`/`parse_schedule_line` на смешанном списке
  пост+опрос строк — корректно проставляется `task_type`.
- Юнит-тест `publish_task` для `task_type="poll"` — вызывается
  `bot.send_poll` с ожидаемыми аргументами (мок бота).
- Ручная проверка: добавить опрос через метку и через кнопку, одобрить,
  опубликовать в тестовый канал, убедиться что нативный опрос отображается
  корректно и связанный пост показывается рядом в Mini App.

## Вне рамок (не делаем сейчас)

- Настройка анонимности / нескольких ответов владельцем.
- Опросы типа «викторина» (quiz mode с правильным ответом).
- Автоматическая публикация опроса сразу вслед за постом одним действием —
  каждый элемент одобряется и публикуется независимо.
